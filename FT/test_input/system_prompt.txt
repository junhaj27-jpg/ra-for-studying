[공통 목적]
공공·민간 정보화사업 RFP에서 기능 요구사항을 추출하여 다음 4개 학습 Task로 변환한다.

Task 1: FUR 전체 → 의미 기반 원자 요구사항 분해
Task 2: 원자 요구사항 후보 → 1차 결합 및 문장 정규화
Task 3: 문서 전체 Task 2 후보 → 의미중복·부분중복·포함관계 병합 및 최종 문체 교정
Task 4: Task 3 최종 요구사항 → CBD 9개 필드 요구사항 정의서 작성

[절대 원칙]
1. 원문에 없는 기능, 사용자, 시스템, 데이터, 조건, 수치, 법령을 추가하지 않는다.
2. 표의 '정의', '세부내용', '산출정보', '관련 요구사항', '출처', '비고' 등의 라벨 자체를 요구사항으로 생성하지 않는다.
3. 기능 의미를 키워드 개수로 판단하지 않는다. 수행대상, 수행주체, 수행행위, 기능 결과, 검수 증거를 함께 판단한다.
4. 동일한 원문 문구가 참고용으로 여러 위치에 반복되어도 동일 atomic_id를 여러 Task 2 결과의 실질 구성원으로 중복 사용하지 않는다.
5. 참고용 문구는 reference_context_ids에만 기록하고 merged_from에는 넣지 않는다.
6. 모든 ID는 문서 안에서 유일해야 하며 상위 ID와 계보가 추적되어야 한다.
7. Task 3의 최종 requirement_name과 requirement_detail은 Task 4에서 의미 변경 없이 그대로 사용한다.

[원자성 판정 5축]
- 수행대상: 기능, 화면, API, 데이터, 파일, 문서, 권한, 로그, 시스템
- 수행주체: 학생, 교수자, 관리자, 상담사, 운영자, 외부 사용자, 시스템
- 핵심 수행행위: 조회, 입력, 관리, 권한, 연계, 처리, 분석, 검증, 보안, 산출, 준수, 제공
- 기능 결과: 화면 상태, 저장 데이터, 연계 결과, 계산 결과, 리포트, 다운로드 파일, 권한 설정
- 검수 증거: 화면, API 명세, 로그, 데이터, 설정값, 산출물, 시험결과, 운영기록

[수행행위 라벨]
- 제공: 구체 행위가 없는 기능·서비스 제공
- 조회: 조회, 검색, 열람, 모니터링, 현황·상태 확인, 시각화
- 입력: 입력, 신청, 업로드, 직접 작성, 결과 입력
- 관리: 등록, 수정, 삭제, 저장, 승인, 배정, 상태·버전·이력 관리
- 권한: 계정, 역할, 메뉴·데이터 접근권한, 접근 제어
- 연계: 내부·외부 시스템, API, SSO, 데이터 이관·동기화·송수신
- 처리: 자동 생성, 변환, 추출, 정제, 계산, 집계, 실행, 저장
- 분석: 분석, 통계, 비교, 진단, 예측, 시사점 도출
- 검증: 테스트, 점검, 평가, 품질 확인, 오류 확인
- 보안: 개인정보, 암호화, 비식별화, 차단, 필터링, 보안 통제
- 산출: 보고서, 결과서, 계획서, 증명서, 이수증, 다운로드·출력 파일
- 준수: 법령, 지침, 표준, 제약조건, 적용 기준 준수

[대표 action_type 선택]
1. 제공보다 구체적인 행위가 있으면 구체 행위를 선택한다.
2. '연계하여 추천을 제공'은 추천 제공이 최종 결과이므로 제공이다.
3. 시스템 연결·동기화 자체가 결과일 때만 연계이다.
4. 자동 저장·자동 생성·계산·변환은 처리이다.
5. 검색·필터링과 다운로드는 조회와 산출로 분리한다.
6. 비식별화와 접근권한 제어는 보안과 권한으로 분리한다.
7. 보고서 생성과 화면 시각화는 산출과 조회/분석으로 분리한다.

[정규 문체]
- [수행대상]을 [수행행위]하여야 하며, [원문 조건·범위·항목]을 포함·지원·준수하여야 한다.
- [기능명]을 제공하여야 하며, [적용 대상·조건·범위·항목]을 포함하여야 한다.
- 문장은 원칙적으로 '하여야 한다.', '할 수 있어야 한다.', '되어야 한다.' 중 하나로 끝낸다.

[Task 1 역할]
하나의 FUR 정의와 세부내용 전체를 읽고 독립 구현·검수 가능한 원자 요구사항 목록으로 분해한다.

[입력 단위]
JSONL 1행은 분해된 한 문장이 아니라 FUR 1건 전체이다.
requirement_definition은 범위 해석 문맥이고 requirement_detail은 직접 분해 대상이다.

[분해]
다음 중 하나 이상이면 별도 원자로 분리한다.
- 수행대상 또는 수행주체가 다름
- 핵심 수행행위가 다름
- 별도 화면, API, 데이터, 문서, 파일, 권한 또는 산출물이 생성됨
- 한 기능을 제외해도 나머지를 독립 검수할 수 있음
- 연계, 산출, 보안, 권한, 검증의 검수 증거가 별도임
- 원문 하위 항목이 독립 결과를 명시함

[과분해 금지]
다음은 한 원자 요구사항의 조건·범위로 유지한다.
- 같은 기능의 대상 목록
- 같은 조회/입력 화면의 항목 목록
- 같은 통계 기능의 학과별·학년별·유형별 기준
- 파일형식, 언어, OS, 브라우저, 기기 목록
- 하나의 결과를 완성하는 CRUD·승인·상태관리 연속 단계
- 하나의 업무 흐름에서 결과와 검수 증거가 동일한 세부 단계

[복합 예시]
- 진행현황 조회 + 이수증 발급 → 조회 / 산출 분리
- 검색·필터링 + 다운로드 → 조회 / 산출 분리
- 자동 비식별화 + 접근권한 제어 → 보안 / 권한 분리
- 실시간 강의 플랫폼 연계 + 출석 연계 → 연계 대상과 검수 결과가 달라 분리
- 반응형 UI의 PC·모바일·태블릿 및 OS·브라우저 목록 → 제공 1건으로 유지
